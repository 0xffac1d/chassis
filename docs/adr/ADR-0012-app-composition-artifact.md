# ADR-0012: App composition artifact

## Context

Chassis governs the individual module — its `CONTRACT.yaml`,
schemas, diagnostics, and assurance posture. It has no vocabulary for
*composition*: the combination of modules that together constitute a
deployable service. The 90-day blueprint refactoring a legacy workspace from 77
crates into 8–12 modules surfaced this gap: module manifests pin the
inside of a module, but nothing pins the *set* of modules, their
revisions, their entrypoints, or their assurance envelope.

Service Weaver was archived June 2025. The published retrospective
attributes the failure to forcing codebase rewrites into
framework-specific component interfaces: adopting the framework
required refactoring existing code into components the framework
understood, which defeated the "just deploy your Go program" value
proposition. Chassis must govern composition **without** imposing a
component model on module internals.

Helm's `dependencies[]` model is authored-only and drift-prone: the
installed chart and the authored manifest disagree silently. Encore's
inferred-only composition model is unreviewable: no artifact to sign,
review, diff, or gate. Backstage's `System` entity captures identity
well but is not a composition artifact — it does not pin revisions or
resolve capability graphs. KubeVela OAM is the right axis but overkill
at module granularity.

## Decision

1. **`App` is a first-class Chassis artifact.** It sits above the
   module manifest and below the deploy target. It is authored as
   `app.yaml` (human intent) and resolved into `app.lock`
   (tool-generated, committed).
2. **`app.yaml` expresses intent.** Fields: `name`, `version`,
   `assurance_level`, `modules[]` (each with `name`, `source`,
   `version_req`), `entrypoints[]` (surface names exported by pinned
   modules), `deploy_targets[]`. Schema at
   `schemas/app/app.schema.json`.
3. **`app.lock` expresses resolution.** Tool-generated from `app.yaml`
   plus the module manifest registry. Fields: resolved rev per module,
   capability fingerprint per module, transitive dep graph, schema
   fingerprint, generation timestamp. Schema at
   `schemas/app/app-lock.schema.json`.
4. **Cargo.toml / Cargo.lock discipline.** `app.yaml` is edited by
   humans and reviewed in PRs; `app.lock` is regenerated by the CLI
   and committed. A lock staler than its manifest fails the app gate.
5. **Modules remain idiomatic Rust crates.** No component interface,
   no framework-specific trait, no `chassis_component!` macro. A
   module is a crate with a `CONTRACT.yaml` and exported capabilities;
   composition is expressed only in the app manifest.
6. **Sub-60-second verification target.** The app gate reads
   `app.lock`, diffs capability fingerprints against current module
   manifests, and returns without invoking `cargo` on the happy path.
   Zero compilation for the fingerprint-only fast path.

## Consequences

- Composition is reviewable: `app.yaml` diffs are small and
  human-meaningful; `app.lock` diffs make rev movement visible.
- Drift is detectable at the app level without rebuilding: the
  capability fingerprint per module is the identity, not the path.
- Modules stay portable: dropping chassis does not require rewriting
  module code, only regenerating the app artifact or walking away
  from it.
- A new app-level gate enters the chassis CI surface; rule IDs are
  pinned above so diagnostics bind to this ADR.
- Consumers who do not compose (library use) never author `app.yaml`;
  the gate is scoped to repositories that declare an app manifest.

## Alternatives considered

- **Helm `dependencies[]`** — authored-only; no resolved artifact
  committed; drift between chart and cluster is the recurring
  operator failure mode.
- **Encore inferred composition** — unreviewable; no artifact for CI
  to gate; forces framework adoption.
- **Service Weaver components** — intrusive component interface;
  archival post-mortem named this as the cause.
- **Backstage `System` entity** — identity-focused catalog metadata,
  not composition-focused; no capability resolution.
- **KubeVela OAM** — correct axis, wrong granularity; overkill at the
  module composition layer.
